# DB Entity ~ Key

### Entity

현실세계에 있는 무언가를 표현 하는 “것”

이 “것”은 추상화된 개념일수도 현실 세계에서 실존하는 물건일 수도 있다.

DB에서는 테이블로 표현된다고 생각할 수 있지만 엔티티는 어디까지나 개념이다.

엔티티(개념) → 릴레이션(논리적 표) → 테이블(물리적 구현)

### Relation

Entity와 헷갈릴 수 도 있지만 정확히 나누면

Entity는 Student, 학생에 대한 개념이고 Relation은 그 개념을 컬럼, 튜플 등으로 표로써

만들어낸 것이 차이점이다.

### Attribute(속성)

Entity 내부에서 꼭 들어 가야 하는 것들이다.

엔티티 내부에서 속성의 의미가 중복될 수 없다.

속성은 우리가 흔히 이해하고 있는 id, name, address를 의미한다.

### Domain

어떤 속성이 가질 수 있는 값들의 집합

좀 더 정확히 설명하면 속성들이 가지고 있는 값들의 처음과 끝

속성이 학점이라면 0.0 ~ 4.5까지를 도메인이라고 말할 수 있다

다만, 현실적인 코드에서는 

```c
credit = models.DecimalField(max_digits=3, decimal_places=1)
```

을 의미하기 때문에 문자열 길이, 데이터 타입으로 의미가 좁아진다.

### 필드와 레코드

쉽게 설명하면 
필드는 하나의 Attribute, 레코드는 DB Attribute 값들의 모음이다(인스턴스)

model.py에서 설명하자면 Attribute들은 변수 값

레코드는 변수 값 안에 들어가 있는 실제 값을 의미한다

어렵게 설명됬다면 단순하게

필드는 변수명, 레코드는 그 밑에 따라오는 행들

### 관계

엔티티와 엔티티 사이의 의미 있는 연결

1 : 1 ⇒ 1대1로 매칭되는 관계

1 : N ⇒ 외래키로써 구분되는 데이터, 

N : M ⇒ 관계 그 자체가 엔티티가 되는 것을 의미한다. FinishLine을 예시로 들자면 user 테이블에서 

MyDonelecture 테이블, Alllecturedata와의 관계이다.

학생이 과목을 “수강한다”에서 수강 자체가 엔티티가 되어 수강 과목들이 나열되어있고

학생 ↔ 수강과목 ↔ 전체 과목의 구조를 이해한다면 좀 더 편할 수 있다.

### Key

의미를 가지고 있는 속성이지 않을까

릴레이션 내부에서 릴레이션 자체를 의미하는 key(PK)

릴레이션간의 관계를 표현하는 key(FK)

후보키, 대체키, 슈퍼키는 실제로 실무에서 쓰이는 개념들은 아니다.

다만 설계 단계에서 하나의 속성이 무엇을 의미하는 지 헷갈리는 것을 막는 역할을 한다고 볼 수 있다.
