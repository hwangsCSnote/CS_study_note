# NW 토폴로지와 병목 현상/성능개선 명령어

> **네트워크 토폴로지**
> 

![스크린샷 2026-02-18 210740.png](%EC%8A%A4%ED%81%AC%EB%A6%B0%EC%83%B7_2026-02-18_210740.png)

노드와 링크가 어떻게 배치되어 있는지에 대한 방식, 연결 형태

**Tree 토폴로지**

계층형 토폴로지, 트리 형태로 배치

노드의 추가/삭제가 쉽고 특정 노드에 트래픽이 집중되면 하위 노드에 영향을 끼침

**Bus 토폴로지**

중앙 통신 회선 하나에 여러개의 노드가 연결되어 공유하는 네트워크 형태

LAN에서 사용되고 설치 비용 감소, 신뢰성 우수, 노드 추가/삭제 쉬움

**Star 토폴로지**

중앙에 있는 노드에 모두 연결된 형태

노드 추가와 에러 탐지가 쉽다

장애 노드가 중앙 노드가 아니라면 영향이 적다

중앙 노드에 문제가 생기면 영향이 크고, 비용이 비싸다

**Ring형 토폴로지**

링처럼 하나의 연속된 길을 통해 형태를 구성

노드의 수가 증가되어도 네트워크상 손실이 없다

노드의 고장을 쉽게 발견할 수 있다

네트워크 구성 변경이 어렵다. 회선에 문제가 생기면 네트워크에 큰 영향을 끼친다

**Mesh형 토폴로지**

그물망처럼 연결된 구조

한 노드에 문제가 생겨도 네트워크를 계속 사용할 수 있다

트래픽도 분산처리 가능하다

노드 추가가 어렵고 비용이 비싸다

**토폴로지가 중요한 이유**

토폴로지란 구성 요소들이 어떻게 연결되어 있는지를 나타내는 구조, 병목 현상을 찾고 해결하는데 중심적인 역할을 한다

네트워크 구조에 따라 데이터 흐름 경로가 다르기 때문에 특정 노드나 링크가 병목 현상을 유발할 수 있다

> **병목현상**
> 

전체 시스템의 성능이 하나의 구성 요소로 인해 제한을 받는 현상

주요 원인

- 네트워크 대역폭 부족
- 서버 처리 능력 저하
- 과도한 트래픽

해결 방안

- 대역폭 확장
- 서버 성능 개선
    - DB 쿼리 튜닝
    - 캐싱 전략 도입
- 지연 시간 감소
    - CDN 활용
- 애플리케이션 최적화
    - 코드 최적화
    - 불필요한 HTTP 요청 제거

> **성능 분석 명령어**
> 

- ping
    
    대상 노드를 향해 패킷을 전송하여 도달하기까지의 시간을 알 수 있음
    
    ```java
    Ping finishline-cku.com [172.239.57.117] 32바이트 데이터 사용:
    요청 시간이 만료되었습니다.
    
    172.239.57.117에 대한 Ping 통계:
        패킷: 보냄 = 4, 받음 = 0, 손실 = 4 (100% 손실),
    ```
    
- netstat
    
    접속된 서비스들의 네트워크 상태 표시
    
    주로 서비스의 포트가 열려 있을 때 사용
    
- tracert
    
    목적지 노드까지 네트워크 경로 확인 용도
    
    목적지 노드까지 구간중 어느 구간에서 느려지는 파악 가능
    
- nslookup
    
    DNS에 관련된 내용을 확인하기 위해 사용
    
    특정 도메인에 매핑된 IP 확인을 위해 사용
    
    ```java
    > nslookup finishline-cku.com
    서버:    finishline-cku.com
    Addresses:  172.234.24.211
              172.239.57.117
    
    DNS request timed out.
        timeout was 2 seconds.
    *** finishline-cku.com에 대한 요청이 제한 시간을 초과했습니다.
    ```
