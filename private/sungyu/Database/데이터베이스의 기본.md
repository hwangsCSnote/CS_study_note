# 데이터베이스의 기본
---

**데이터베이스**(Database) 란?
: 데이터의 집합을 의미

사람들은 데이터베이스를 관리하기 위해 **DBMS**(DataBase Management System)를 사용한다.
ex) Oracle, MySQL, MSSQL etc

그렇다면 DBMS를 **’ 왜 ’** 사용할까?
-> 만약 우리가 데이터를 파일로 관리한다면, 데이터가 많아질 수록 원하는 데이터를 찾거나 관리하기에 
어려움이 생길 것이다.

### 엔터티(Entity)
: 실체, 본체, 개체
---
> 여러 **속성**을 가진  명사/개체.


![](%E1%84%83%E1%85%A6%E1%84%8B%E1%85%B5%E1%84%90%E1%85%A5%E1%84%87%E1%85%A6%E1%84%8B%E1%85%B5%E1%84%89%E1%85%B3%E1%84%8B%E1%85%B4%20%E1%84%80%E1%85%B5%E1%84%87%E1%85%A9%E1%86%AB/Gemini_Generated_Image_aqb6f2aqb6f2aqb6.png)<!-- {"width":184} -->
예를 들어
**회원 엔터티**에는 **이름**, **아이디**, **주소**, **전화번호** 라는 속성이 존재한다.

-> 속성은 서비스의 요구 사항에 따라 다르게 구성될 수 있으며, **속성이 없는 엔터티는 존재할 수 없다.**




### 릴레이션(Relation)
: 관계
---
> 데이터베이스에서 정보를 저장하는 기본 단위  ex) 테이블(SQL), 컬렉션(NoSQL)



> [!NOTE] SQL vs NoSQL
> - **SQL**(Structured Query Language)
> - **NoSQL**(Not Only Structured Query Language)
> 
> - **SQL 데이터베이스**는 테이블(Table) 구조로 데이터를 저장하고 관리한다.
>   -> 일관성을 중시
> - **NoSQL 데이터베이스**는 문서, Key-Value 등 유연한 구조로 데이터를 저장하고 관리한다. -> 유연성, 확장성, 성능을 중시

### 속성(Attribute)
---
> 릴레이션에서 관리하는 구체적이며 고유한 성질, 특징을 의미 
	-> **컬럼으로 표현할 수 있는 단위**

	ex) **차 엔터티**에는 **번호**, **바퀴 수**, **색**, **차종** 등의 특징이 속성이 될 수 있다.

- 한 개의 엔터티는 2개 이상의 속성을 가진다.
  -> 하나의 테이블은 두 개 이상의 컬럼으로 구성된다.
- 하나의 속성에 여러 개의 값이 있는 다중값일 경우, 별도의 엔터티를 이용하여 분리한다.



### 도메인(Domain)
: 영역, 분야, 범위
---
>릴레이션에서 각각의 속성들이 가질 수 있는 값의 범위 (값의 범위, 데이터 타입, 제약사항 등)

ex) **회원 릴레이션**에서 성별에 대한 속성이 있다고 가정.
이를 `남자 = 0`, `여자 = 1` 로 값을 사용하기로 정의했다면, 
0 과 1 외의 값은 사용할 수 없게 정의한 범위를 **도메인**이라고 생각하면된다.
추가로 타입과 사이즈를 정해주는 것도 도메인에 포함된다.


**< 회원 >**
| 이름  | 성별  | …   |
|:---:|:---:|:---:|
| …   | 1   | …   |
| …   | …   | …   |



### 필드(Field)
---
> 데이터베이스 테이블의 열을 뜻하며, 하나의 필드는 특정 속성(Attribute)에 해당

ex) 학생 엔터티 (학번, 이름, 전공, …)

**< Student >**
| 학번(Student ID) | 이름(Name) | 전공(Major) | …   |
|:--------------:|:--------:|:---------:|:---:|
| 필드             | 필드       | 필드        | 필드  |

- 학생 엔터티는 Student 라는 테이블로 학번, 이름, 전공을 속성으로 가진다.
- 이때 필드는 Student ID, Name, Major 등 특정 속성(Attribute) 을 뜻한다.



### 레코드(Record)
: 기록
---
> 데이터베이스 테이블에 쌓이는 행(Row) 단위의 데이터

ex)
**< 학생 >**
| 학번(Student ID) | 이름(Name) | 전공(Major) | …   |
|:--------------:|:--------:|:---------:|:---:|
| 1              | 김철수      | 소프트웨어     | …   |
| 2              | 이영희      | 간호학       | …   |
| 3              | 박지훈      | 건축공학      | …   |

- 학생 테이블에서 `1 | 김철수 | 소프트웨어 | … ` 는 하나의 레코드를 의미한다.
- 하나의 레코드는 여러 필드로 구성된다.
- 튜플(Tuple)이라고도 한다.



### 관계(Relationship)
---
> 엔터티간 연관성을 나타낸 개념

- 관계를 정의할 때는 인스턴스(각 행 데이터) 간의 논리적인 연관성을 파악하여 정의한다.
- 엔터티를 어떻게 정의하느냐에 따라 변경되기도 한다.

**관계 차수**(Cardinality)
: 한 엔터티의 레코드(인스턴스)가 다른 엔터티의 레코드(인스턴스)와 어떻게 연결되는지를 나타내는 표현
- **1:1 관계**
  ex) 유저 한 명당 이메일 한 개씩 갖고 있는 경우
- **1:N 관계**
  ex) 한 유저가 장바구니에 여러 개의 상품을 넣는 경우
- **N:M 관계**
  ex) 한 명의 학생은 여러 개의 강의를 수강할 수 있고, 강의도 여러 명의 학생을 포함하는 경우



### 키(Key)
---
> 하나의 엔터티에 구성된 여러 개의 속성 중에서 엔터티를 대표할 수 있는 속성

#         📜                                      📊
###        논리 모델링                      vs                       물리 모델링

| 논리 모델링 | vs  | 물리 모델링 |
|:------:|:---:|:------:|
| 식별자    |     | 키(Key) |

ex) **학생 엔터티**의 주 식별자는 `학번` 속성  **vs**  **학생 테이블**의 기본키는 `학번` 컬럼


**기본키**(Primary Key, PK)
- 테이블을 대표할 수 있는 키
- 유일성과 최소성을 만족하는 키

**후보키**(Candidate Key)
- 기본키가 될 수 있는 후보를 가리킨다.
- 후보키들 중 하나가 기본키가 되고, 나머지를 대체키라고 부른다.
- 유일성과 최소성을 만족하는 키

**대체키**(Alternate Key)
- 여러 후보키 중 기본키가 아닌 키

**외래키**(Foreign Key, FK)
- 다른 테이블의 기본키를 참조하는 키

**슈퍼키**(Super Key)
- 고유하게 식별할 수 있는 모든 조합
- 유일성은 만족하지만 최소성은 만족하지 않는 키
  ex) 학생 테이블에서 학번으로만 PK를 구성할 수 있지만, `학번 + 이름`으로 구성하는 경우

**인조키**(Artificial Key)
- 인위적으로 생성한 키
- 자연적으로 존재하는 속성(자연키)이 없을 때 기본키로 인조키를 사용한다.
