# 데이터베이스 종류  

### 관계형 데이터베이스(Relational Database, RDB)  
> 행(Row)과 열(Column)을 가지는 테이블 형식으로 데이터를 저장하는 데이터베이스  

<br>

**관계형 데이터베이스의 특징**  
- 정형화된 스키마  
- 트랜잭션 처리에 적합  
- 수직적 확장(scale-up)에 유리 : 하나의 서버 사양을 높여 성능을 업그레이드  
- 정규화, 관계를 통해 데이터 중복을 최소화  
- 데이터 무결성을 보장, 높은 신뢰성

**쿼리 파서**(Query Parser)  
- 사용자의 쿼리문을 토큰으로 분리하여 트리 형태의 구조로 변경  
- 문법 오류 여부 확인  

**전처리기**(Preprocessor)  
- 예약어를 제외한 토큰이 DB에 존재하는지, 접근 권한이 있는지 확인  

**옵티마이저**(Optimizer)  
- Parse 트리(Tree)를 구체적인 실행 계획으로 변환  
- 쿼리를 최적으로 실행하기 위한 실행 계획을 수립 (비용 기반 최적화, 규칙 기반 최적화)  

**스토리지 엔진**(Storage Engine)  
- 실행 엔진의 요청을 처리하기 위해 디스크로부터 데이터 읽기/쓰기

### NoSQL(Not only SQL) 데이터베이스  
> 키(Key) - 값(Value) 형태로 데이터를 저장하는 데이터베이스  

<br>

**NoSQL 의 특징**  
- 비정형 데이터 활용 가능  
- 스키마의 유연성  
- 데이터 중복 허용, 일관성 보장 X  
- 수평적 확장(scale-out)에 유리 : 여러 서버를 추가로 설치  

<br>
<br>

**MongoDB**  
> 컬렉션(Collection) 단위로 데이터를 관리하는 도큐먼트 기반의 데이터베이스  

<img width="290" alt="image 2" src="https://github.com/user-attachments/assets/4ce79048-e193-443a-be7e-8103666b61b6" />

**특징**  
- JSON을 통해 데이터에 접근, BSON(Binary JSON) 형태로 데이터를 저장  

  <img width="478" alt="image 3" src="https://github.com/user-attachments/assets/06c9d895-ab83-412d-b7eb-c353153cbcc5" />

- 도큐먼트(Document) 생성 시, 다른 컬렉션과 구별되는 유니크한 값 (ObjectID)가 생성되어 함께 저장  
- 조인 연산이 필요없는 구조로, 읽기/쓰기 속도가 빠름  
- 같은 데이터를 여러 서버에 복제하여, Primary 서버 장애 시, Secondary 서버 자동 승격 (Failover)  
  -> 서버 장애에 대비, 높은 안정성  

<br>
<br>

**redis**  
> 인메모리 데이터베이스(In-Memory Database)이자, 키(Key) - 값(Value) 기반 데이터베이스  

<img width="345" alt="image 4" src="https://github.com/user-attachments/assets/fd1f6f4c-ed15-4d40-a99b-9fb856177286" />


**특징**  
- 데이터를 하드 디스크가 아닌 메모리(RAM)에 저장하기 때문에 속도가 빠름 (하드 디스크에 백업)  
- 싱글 스레드 방식으로 한 번에 하나의 명령을 처리하기 때문에 락(Lock) 없이 빠른 처리가 가능  
- TTL(Time-To-Live) 기능으로 키(Key) 마다 유효 시간을 설정하여 세션 관리, 캐시 만료 등 기능 활용 가능  

<br>

### RDBMS vs NoSQL  
- **스키마** (구조적 vs 유연함)  
- **단위** (테이블 vs 컬렉션 / 튜플 vs 도큐먼트)  
- **확장성** (수직적 vs 수평적)  
- **활용** (정확도 우선 vs 입출력 속도 우선)
