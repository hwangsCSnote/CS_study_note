### 선언형과 함수형 프로그래밍  

**함수형 프로그래밍**(Functional programming)  
> 입출력에만 의존하는 ‘순수 함수’들을 블록처럼 쌓아 로직을 구현하거나,  
> 다른 함수의 값을 매개변수로 받아 로직을 생성하는 ‘고차 함수’를 통해 재사용성을 높인 프로그래밍 기법.  

ex) 배열에서 최댓값을 찾으려는 상황  
	- reduce(<콜백 함수>, <초기값>)  
```javascript
const numbers = [1, 2, 3, 4, 5, 11, 12];
const ret = numbers.reduce((max, num) => num > max ? num : max, 0);
console.log(ret);  // 12
```

- reduce( ) 고차 함수를 사용  

<br>

**순수 함수**  
> 입력 값에 의존해서 출력값을 구성하는 함수  
```javascript
const pureFunc = (a, b) => {
	return a + b
}
```

**객체지향 프로그래밍**(Object Oriented Programming)  
> 객체들의 집합으로 프로그램의 상호 작용을 표현하며, 객체 내부에 선언된 메서드를 활용하는 방식  

- 설계에 많은 시간이 소요된다.  
- 처리 속도가 다른 프로그래밍 패러다임에 비해 상대적으로 느리다.  

ex) 배열에서 최댓값을 찾으려는 상황  
```javascript
const numbers = [1, 2, 3, 4, 5, 11, 12];

class List {
	constructor(list) {
		this.list = list
		this.mx = list.reduce((max, num) => num > max ? num : max, 0)
	}
	getMax() {
		return this.mx
	}
}

const ret = new List(numbers)
console.log(ret.getMax());  // 12
```

**특징**  
- **추상화**(Abstraction) : 복잡한 시스템으로부터 핵심적인 개념, 기능만을 간추려내는 것  
- **캡슐화**(Encapsulation) : 객체의 속성과 메서드를 하나로 묶고 일부를 외부에 감춰 은닉하는 것  
- **상속성**(Inheritance) : 상위 클래스의 특성을 하위 클래스에서 이어받아 재사용하거나 확장하는 것  
- **다형성**(Polymorphism) : 하나의 메서드나 클래스가 다양한 방법으로 동작하는 것  
  - 오버로딩(Overloading) : ~같은 이름의 메서드~를 여러 개 두는 것 (매개변수에 차이가 있도록)  
  - 오버라이딩(Overriding) : 상위 클래스로부터 상속받은 메서드를 하위 클래스에서 ~재정의~ 하는 것  

<br>

**설계 원칙** (SOLID 원칙)  
- **단일 책임 원칙** (Single Responsibility Principle, SRP)  
  - 모든 클래스는 각각 하나의 책임만 가져야 하는 원칙  
- **개방 - 폐쇄 원칙** (Open Closed Principle, OCP)  
  - 유지 보수 사항이 생긴다면 코드를 쉽게 확장 가능해야하며 기존 코드를 변경하지 않아야 한다.  
- **리스코프 치환 원칙** (Liskov Substitution Principle, LSP)  
  - 객체는 프로그램의 정확성을 유지하면서 하위 타임의 인스턴스로 바꿀 수 있어야 한다.  
  - 부모 - 자식 클래스 계층관계에서 부모 객체에 자식 객체를 넣어도 시스템이 문제 없이 돌아가야 한다.  
- **인터페이스 분리 원칙** (Interface Segregation Principle, ISP)  
  - 하나의 일반적인 인터페이스보다 구체적인 여러 개의 인터페이스를 만들어야 한다.  
- **의존 역전 원칙** (Dependency Inversion Principle, DIP)  
  - 자주 변하는 것들은 추상화된 인터페이스나 상위 클래스를 두어 변화에 영향받지 않도록 하는 원칙을 말한다.  
  - 상위 계층은 하위 계층의 변화에 대한 구현으로부터 독립해야 한다.

### 절차형 프로그래밍  
> 로직이 수행되어야 할 연속적인 계산 과정으로 이루어진 방식  

- 일이 진행되는 방식으로 코드를 구현하면 되기 때문에 코드 가독성이 좋으며 실행 속도가 빠르다.  
- 계산이 많은 작업 등에 사용된다. (대기 과학 관련 연산 작업, 머신러닝의 배치작업 등)  
- 모듈화하기 어려우며, 유지 보수성이 떨어진다.  

<br>

ex) 배열에서 최댓값을 찾으려는 상황  
```javascript
const numbers = [1, 2, 3, 4, 5, 11, 12];

let ret = 0

for (let i = 0; i < numbers.length; i++) {
	ret = Math.max(numbers[i], ret)
```

### 패터다임의 혼합  
- 비즈니스 로직이나 서비스의 특징을 고려해서 패러다임을 정하는 것이 좋다.  
- 여러 패러다임을 조합하여 상황과 맥락에 맞는 패러다임 간의 장점만 취해 개발하는 것이 좋다.  
- 하나의 서비스에 백엔드 머신러닝 파이프라인과 거래 관련 로직이 있다면, 머신러닝 파이프라인은 절차지향형 패러다임, 거래 관련 로직은 함수형 프로그래밍을 적용하는 것이 좋다.
